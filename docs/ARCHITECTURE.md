Sitelayer is a multi-tenant construction operations monorepo with a React single-page app, a native Node.js HTTP API, a background worker, and shared TypeScript packages. Startup is tier-aware: local, dev, preview, and prod deployments validate their database and blueprint-storage bindings so the wrong environment cannot silently point at production data.

The web app boots through `main.tsx`, installs Clerk for non-fixture sessions, defers Sentry and web-vitals work until idle, and renders the SPA shell from `App.tsx`. The signed-in shell loads the current session, bootstrap data, company list, sync status, and QBO connection state for the active company, while the client API layer attaches the company slug and Clerk token to each request, caches GET responses, and queues failed mutations for offline replay. Route views are lazy chunks with nav hover/focus preloads, and the bootstrap includes one-shot reload recovery for stale chunk URLs after a deploy. The top-level routes cover Confirm Day, Clock, Schedule, Projects, Takeoffs, Estimates, Integrations, Onboarding, Rentals, Bonus Simulation, and an admin-gated Audit view, with a non-production Dev scratch surface available for local inspection. Projects, takeoffs, and estimates stay tightly linked so changes to measurements, schedules, labor, and blueprints flow back into the selected project summary without leaving the app shell.

The API is a framework-free HTTP server that owns routing, JSON parsing, CORS, auth, metrics, and audit recording. Identity resolution prefers Clerk JWTs, falls back to an internal bearer token for service traffic, and only allows header-based fallback when the tier permits it. Every resource lookup is company-scoped through the company slug or id plus membership checks. The API exposes bootstrap/session/features/version/metrics endpoints, CRUD for projects, customers, workers, divisions, pricing profiles, bonus rules, service items, labor entries, material bills, schedules, takeoff measurements, and blueprints, plus company creation and membership management. Company creation seeds the LA template’s divisions, service items, pricing profile, and bonus rule defaults; QBO endpoints handle OAuth, mappings, connection state, and sync triggering; and queued integration work is written to mutation outbox and sync-event tables. Blueprint files are stored through either the local filesystem or DigitalOcean Spaces, and debug-trace lookups can fetch Sentry trace data for a request or trace id under rate limiting.

The worker is a single-company poller that checks pending outbox and sync-event counts, claims work with skip-locked transactions, applies batches, and refreshes integration connection timestamps after successful processing. Shared packages hold the LA seed template and the cost, geometry, margin, and bonus calculations; the tier-aware config loader; structured logging with request-context propagation; and the queue processor used by both the API and worker. Together these pieces keep the frontend state, API mutations, and background sync path aligned around the same company-scoped database model.
