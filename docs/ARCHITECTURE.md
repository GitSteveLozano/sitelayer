Sitelayer is a multi-tenant construction takeoff and estimation system composed of a React single-page frontend, a Node.js HTTP API, a background worker, and shared TypeScript packages. The system supports four deployment tiers — local, dev, preview, and prod — with tier-aware configuration enforced at startup; mismatched database names or storage buckets cause the process to refuse to start rather than silently operate against the wrong environment.

The web frontend is a React application bundled with Vite and routed with React Router, styled with TailwindCSS and Radix UI primitives. On load the application shell fetches a bootstrap payload containing the full reference catalogue: companies, divisions, service items, customers, workers, pricing profiles, bonus rules, and integration mappings. Project-level data — blueprints, measurements, schedules, material bills, and labor entries — is fetched lazily when the user selects a project. The frontend implements offline resilience through a client-side mutation queue stored in localStorage. When a mutation fails due to network absence it is queued and replayed on reconnect or on a 15-second polling timer; 409 conflict responses are retained in the queue for manual resolution. A non-production environment ribbon drawn from the features endpoint signals the active tier to users.

The API is a plain Node.js HTTP server backed by PostgreSQL. Routes are dispatched by method and URL prefix within a single request handler, with CORS and JSON body parsing handled inline. Multi-tenancy is expressed per-request through company-slug and user-id headers rather than session tokens. Mutations that need deferred processing — such as entities that synchronize to QuickBooks Online — are written to a `mutation_outbox` or `sync_events` table, keeping the synchronous response path fast. Blueprint documents are stored through a pluggable storage adapter: local filesystem in the local and dev tiers, and DigitalOcean Spaces (S3-compatible, served by Minio in Docker Compose) in preview and prod. A rate-limited, HMAC-secured debug endpoint surfaces raw Sentry trace data alongside outbox and sync-event rows for a given trace ID or request ID.

The background worker is a separate Node.js process that polls both the outbox and sync-events tables on a configurable interval (default 10 seconds). On each tick it checks for pending or stale-processing rows for a single `ACTIVE_COMPANY_SLUG`, claims a batch using `SELECT ... FOR UPDATE SKIP LOCKED` inside an explicit transaction, marks each row applied, and updates `integration_connections.last_synced_at`. Sentry spans are emitted for each applied row, linking worker activity back to the originating API trace. Scoping one worker process to one company slug allows independent worker instances to serve isolated tenants.

Shared packages encapsulate cross-cutting concerns. The domain package provides pure functions for polygon geometry calculations, project costing, margin analysis, and bonus payout rules. The config package loads tier and feature-flag configuration from environment variables with strict validation and exposes `postgresOptionsForTier`, which injects `app.tier` as a PostgreSQL session variable so row-level security policies can inspect the active tier. The logger package provides structured, context-aware logging with request-scoped correlation IDs. The queue package implements the `FOR UPDATE SKIP LOCKED` outbox protocol and exposes both a pool-based and a client-based API, allowing the server and worker to share identical processing logic while managing their own connection lifecycles. Sentry instruments the API, worker, and web frontend end-to-end, with trace context propagated through `sentry-trace` and `baggage` headers across process boundaries.
