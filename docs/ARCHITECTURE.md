Sitelayer is a multi-tenant construction takeoff and estimation system built as a monorepo containing a React single-page frontend, a plain Node.js HTTP API, a background worker, and shared TypeScript packages. The system enforces strict tier-aware configuration across local, dev, preview, and prod environments, verifying database and storage bindings to prevent cross-environment contamination.

The web frontend is organized around four primary views—Projects, Takeoffs, Estimates, and Integrations—rendered through React Router with TailwindCSS and Radix UI for styling. On startup the app fetches a bootstrap payload and a session in parallel, populating reference data such as divisions, service items, customers, and pricing profiles. Project-specific entities including blueprints, measurements, material bills, and schedules are loaded lazily as the user navigates to the Takeoffs view. To handle network disruptions, the frontend queues mutations locally and runs a background replay loop every 15 seconds, also triggering a replay whenever the browser comes back online. An environment ribbon surfaces the current tier and feature flags fetched from a dedicated API endpoint; a dev-only scratch route is hidden on production tiers.

The Node.js API server processes requests directly with the native HTTP module, performing routing, JSON parsing, and CORS without a web framework. Identity is resolved in layers: Clerk JWT signatures are verified first; if no Clerk key is configured, an internal bearer token is accepted; in non-production tiers a request-header fallback is permitted. Multi-tenancy is enforced through the `x-sitelayer-company-slug` header, and all resource access is scoped to the resolved company. Companies may be created directly through the API: the creation endpoint validates the slug format, provisions a new company record, assigns the requesting user as an admin member, and optionally seeds the company with canonical division, service item, pricing profile, and bonus rule defaults drawn from the domain package. Company memberships carry one of four roles—admin, member, foreman, or office—and an admin-only endpoint allows adding or updating members. A Clerk webhook endpoint (`POST /api/webhooks/clerk`) accepts svix-signed lifecycle events for user creation, update, and deletion; current handlers are intentionally minimal and preserve the audit trail rather than cascading deletes. The API reads and writes a PostgreSQL database synchronously for most operations but defers integration-heavy work—principally QuickBooks Online synchronization—to mutation outbox and sync events tables. It also records audit logs for entity changes. Blueprint storage is abstracted behind an adapter targeting the local filesystem or DigitalOcean Spaces depending on the tier. A separate debug-trace endpoint lets authorized clients retrieve Sentry trace data for a given request ID, gated behind HMAC authorization and a token-bucket rate limiter.

The background worker polls the outbox and sync events tables on a configurable interval, first querying pending counts before committing to a batch. When items are present it claims them inside a skip-locked transaction to avoid contention, processes the batch, and on success updates the `integration_connections` table to reflect the latest sync timestamp. Each worker instance is scoped to a single company slug, enabling isolated horizontal scaling.

Shared TypeScript packages encapsulate cross-cutting concerns: the domain package defines project cost and margin calculations, polygon geometry for takeoff area measurement, bonus payout logic, and the canonical tenant seed template; the config package enforces tier-aware environment validation at startup; the logger package provides structured logging with request-context propagation; and the queue package implements the outbox and sync-event processing logic shared by both the API and the worker. The entire system carries end-to-end Sentry instrumentation across the browser, API, and worker, and the API additionally exposes Prometheus metrics covering request latency, database pool state, and queue depth.
