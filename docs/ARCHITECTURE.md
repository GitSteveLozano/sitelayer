Sitelayer is a multi-tenant construction takeoff and estimation system built as a monorepo containing a React single-page frontend, a plain Node.js HTTP API, a background worker, and shared TypeScript packages. The system enforces strict tier-aware configuration across four deployment environments (local, dev, preview, and prod), validating that database connections and storage buckets align with the active tier to prevent accidental cross-environment contamination.

The web frontend operates as a resilient, offline-capable React application. It uses React Router for navigation, and TailwindCSS alongside Radix UI for component styling. When the application loads, it fetches a comprehensive bootstrap payload containing reference data such as companies, service items, and pricing profiles. Project-specific data like blueprints, schedules, and material bills are fetched lazily upon project selection. To ensure continuous operation during network disruptions, the frontend queues mutations in local storage. Once connectivity is restored, an offline synchronization loop replays these queued mutations to the backend, preserving conflicts for manual resolution. A non-production ribbon visually indicates the active environment tier.

The Node.js API server manages all incoming requests directly via the native HTTP module without a heavy framework. It dispatches routes, handles JSON parsing, and enforces cross-origin resource sharing (CORS). Multi-tenancy is established through explicit request headers representing the company slug and user ID, avoiding complex session states. The API communicates with a PostgreSQL database, executing synchronous reads and writes while deferring expensive or integration-heavy operations, such as QuickBooks Online synchronization, to a mutation outbox and sync events table. Blueprint document storage is abstracted behind an adapter that targets the local filesystem for development or DigitalOcean Spaces for preview and production environments.

A robust background worker system built on a shared queue package continuously polls the outbox tables. Using a skip-locked transaction model, it securely claims batches of deferred events to process them without locking contention. Each worker instance scopes its processing to a specific company slug, enabling isolated and scalable background processing per tenant.

Shared TypeScript packages encapsulate the system's core cross-cutting concerns. The domain package centralizes business logic for project costing and margin analysis. The config package securely parses environment variables and enforces tier validation. The logger package supplies structured logging with request-scoped correlation. Finally, the entire system is instrumented with Sentry, propagating trace context end-to-end from the browser through the API and into the background workers, maintaining complete visibility across process boundaries.
